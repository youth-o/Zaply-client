## 배경 및 필요성

### 1️⃣ 문제정의

```어떤 분석을 통해 문제를 인식했나요?```

크리에이터 경제가 빠르게 성장함에 따라, 유튜브·틱톡·인스타그램 등 **다양한 SNS를 동시에 운영하는 1인 크리에이터**들이 급증하고 있습니다.

![1](https://github.com/user-attachments/assets/dce62510-b4c2-4676-a016-a30f082f8fd0)

*자료 출처: 센터타워*

그러나 **각기 다른 플랫폼 포맷에 맞춰 콘텐츠를 반복 제작하고**, 성과는 플랫폼별로 나뉘어 **별도로 분석·관리해야 하는 비효율**을 겪고 있다는 점이 리서치를 통해 확인되었습니다.

> 👉🏻 **한줄 문제정의**
> 
> 멀티플랫폼(SNS) 운영 환경에서 **콘텐츠 반복 제작**, **분산된 성과 데이터**, **비효율적인 주제 선정**으로 인해 크리에이터가 콘텐츠 운영 전반에 **과도한 시간과 노력을 소모**하고 있습니다.

### 2️⃣ 문제인식

```설정한 가설과 검증과정, 이를 바탕으로 문제를 어떻게 정의했나요?```

재플리 팀은 다음과 같은 가설을 바탕으로, 다양한 카테고리의 크리에이터 6인과 심층 인터뷰를 진행하여 문제를 구체화했습니다.

- **가설1**: 크리에이터는 동일한 주제를 플랫폼마다 따로 제작하는 데 시간과 노력을 많이 소비한다.
- **가설2**: 플랫폼별로 흩어진 성과 데이터를 통합 분석하지 못하고 있다.
- **가설3**: 콘텐츠 주제 기획 시 데이터 기반 인사이트가 부족해 시행착오가 잦다.

### 📌 인터뷰 종합 인사이트 요약

**공통 Painpoint**
- 시간 부족 및 반복 작업 피로감
- 다중 플랫폼 관리(업로드·댓글·분석)의 비효율
- 빠르게 변하는 트렌드와 알고리즘 대응 어려움

**기능별 주요 painpoint**
- 콘텐츠 재가공 (워딩/비율/톤 재편집)
- 인사이트 개별 확인의 번거로움
- 댓글 누락 및 반복 대응 스트레스
- 알고리즘 불확실성에 따른 노출 한계

> 🎯 **결론**
> 
> 이처럼 단순 반복 작업, 흩어진 관리 환경, 복잡한 플랫폼 구조는 **크리에이터의 지속 가능한 활동을 가로막는 병목**이 되고 있습니다.

### 3️⃣ 해결 방안

```문제를 어떻게 해결해 나가고자 하나요?```

> *"하나의 콘텐츠로, 모든 플랫폼에."*  
> zaply 슬로건

재플리는 이러한 문제를 해결하기 위해 **다중 콘텐츠 발행 및 관리 AI 플랫폼**을 개발하고 있습니다.

#### 1. 콘텐츠 입력 및 변환

**⇒ 하나의 콘텐츠를 입력하면, 모든 플랫폼에 맞춰 자동 변환됩니다.**

**콘텐츠 입력**
- 텍스트·제목·이미지 업로드로 간편하게 시작
- 이미지 최대 10장 업로드 가능

**AI 콘텐츠 변환**
- 미리 설정된 프롬프트 기반으로 플랫폼 맞춤 변환
- 인스타그램, 스레드, 페이스북 등 형식 자동 최적화

**플랫폼별 미리보기 및 수정**
- 변환 결과를 한 눈에 확인
- 필요시 해시태그, 워딩, 이미지 순서 등을 수정 가능 (1차 MLP에서는 일부 구현)

#### 2. 인사이트 통합 관리

**⇒ 모든 플랫폼의 게시물 성과 데이터를 한눈에 확인하고 분석할 수 있습니다.**

**성과 데이터 수집 및 시각화**
- 게시물별 반응을 테이블 형태로 일괄 제공

#### 3. 댓글 목록 표시 및 관리

**⇒ 모든 플랫폼들의 댓글을 하나의 창에서 모두 관리할 수 있습니다.**

**댓글 수집 및 정렬**
- 주기적으로 댓글 수집
- 플랫폼, 작성시간, 상태별로 필터링 가능

**댓글 응답 지원**
- 유사 질문 자동 감지 및 OpenAI 기반 답변 제안
- 템플릿 선택 또는 직접 수정하여 응답 가능

**즉시 게시 기능 제공**
- 수정 완료된 댓글을 플랫폼 API로 바로 게시

### 🧠 포지셔닝

#### ✅ 차별화 포인트

**AI 기반 콘텐츠 자동 변환 및 멀티 플랫폼 배포까지 자동화**
- GPT 기반 텍스트 변환 + 플랫폼별 맞춤 프롬프트 세팅으로, 콘텐츠 제작부터 배포까지 한 번에
- Instagram, Threads, Facebook 등 주요 SNS API 연동 및 자동 배포 기능 제공

**크리에이터 중심의 사용 흐름과 UX 설계**
- 제목·텍스트·이미지 입력만으로 콘텐츠 완성 → AI 자동 변환 → 플랫폼별 미리보기·수정
- 댓글 통합 목록 관리 + AI 응답 제안 기능으로 피드백 대응 효율화
- 성과 통합 대시보드로 모든 콘텐츠의 조회수, 좋아요, 저장 등 주요 지표를 시각화 제공

#### 🚀 기회요소

- 국내 시장에 **크리에이터 전용 콘텐츠 자동화 SaaS**가 본격 출시되지 않음
- 콘텐츠 운영에 필요한 **다중 플랫폼 관리 수요는 급증**, 반복 제작·배포의 피로도도 증가
- 실제 인터뷰 결과, 미러링 부담, 시간 부족, 인사이트 분석 어려움, 댓글 대응의 비효율성 등이 공통 문제로 도출됨
  → 재플리는 이 문제를 **단일 플랫폼에서 자동화 + 통합 관리로 해결**

```답변```
- 그 플랫폼에 맞춘 감성으로 글을 작성하는 과정에서 고민이 많이 되었다.
- 플랫폼 성격에 맞도록 말투나 정보를 약간씩 바꿔서 입력해야하는 경우가 불편합니다.
- 적은 내용이 날아가서 다시 동일한 내용을 작성해야하는 것..
- 콘텐츠에 대한 내용을 적는 것이 가장 어려웠습니다
- 로그인 여러번하는 번거로움
- 동일한 내용을 여러번 작성해서 올려야 하는 것
- 각 플랫폼마다 느낌이 달라서 다르게 작성하는게 불편
- 이미지 올리기?

- 댓글바로가기 = 해당 플랫폼에서 유도하려는 기능이 아니면 맨 아래에 있는게 좋을듯 댓글관리 위에 있는 것 보다
- 하단 CTA가 고정되어있어서 영역을 가리는 것 + 이미 나와있는 것 = 오히려 이해하기 어려운 느낌
- 어려웠다. 긍정적/부정적 기준에 대한 이해가 모호함. 전체 댓글 보기 있어야 할듯.
- 진입화면에서 3개까지만 제한을 두지 말고 오늘,어제 정도는 개수 제한 더 늘려도 좋지않을지?
- 디바이더 어플별 차이가 잘 안느껴짐
- 댓글바로가기 밑에 나오는게 좋을듯.
- 같이 묶어서 대답할만한거 한번에 대답하는거 좋은듯
- 정보의 양이 너무 많음, 점 더 여백을 많이 주어야 좋을듯
- 프라이머리 컬러가 잘 안 보임
- 댓글을 눌러야 겠다는 생각이 안듦
- 문의,긍정,부정 댓글 간격띄우기
- 댓글 숨기기 후 댓글 복구하기가 어려웠음
- 부정 탭 들어가면 댓글 숨기기 누르기 전 오른쪽을 먼저 클릭함
- 비슷한 댓글 찾는 것에 초기 어려움 있었음
- AI 답변 생성 기대감 및 이해도: 잘 생성이 되고 좋았음. 그런데 AI 사용하는 부분이 많은데 이게 잘 작동이 될지가 의문
- 부정 탭을 바로 인지하지 못함
- 댓글 숨기기 시 오른쪽 버튼을 먼저 클릭함 (수민과 공통된 액션)
- 부정 탭을 바로 찾지 못함
- 비슷한 답글에 대한 일괄 답변을 하나 이상 눌러야 된다는 설명이 있으면 좋을 듯함
- 숨기기는 좋았으나 복구하는 버튼은 선택할 수 있다라는 것을 인지하기 어려울 수 있음 복구 버튼이 앞에 있는게 특이함

## Commit Convension

커밋 메시지는 `태그: 커밋 메시지` 형식으로 작성합니다.
> 예시: `git commit -m "feat: 카카오 로그인 기능 구현"`

| 태그명 | 설명 |
|--------|------|
| **feat** | 새로운 기능 개발 |
| **fix** | 코드 및 버그 수정 |
| **style** | CSS 스타일링 및 퍼블리싱 작업 |
| **docs** | 문서 작업(README.md 등) |
| **test** | 배포 테스트, QA 테스팅 관련 |
| **chore** | 패키지 매니저 수정, 그 외 기타 수정 |
| **build** | 빌드 시스템 또는 외부 종속성에 영향을 미치는 변경 사항 |
| **refactor** | 코드 리팩토링 |
| **rename** | 폴더 혹은 파일명 변경 |
| **remove** | 파일 삭제 |

## ⭐️ 기술스택/선정이유
### 1️⃣ Java 21

- Java 21은 최신 언어 기능(예: 패턴 매칭, 레코드, 가상 스레드 등)을 제공하여 코드의 가독성과 유지보수성을 높이며, 개발 생산성을 향상시킵니다.
- 최신 버전의 자바는 성능 최적화와 효율적인 메모리 관리 기능이 개선되어, 대규모 애플리케이션에서도 안정적이고 빠른 실행이 가능합니다.
- 장기 지원 버전이므로, 앞으로의 유지보수와 안정성 측면에서 신뢰할 수 있는 기반을 제공합니다.

### 2️⃣ SpringBoot 3.3.10

- 최신 버전의 Spring Boot는 스프링 프레임워크 및 관련 라이브러리와의 호환성이 뛰어나며, 보안 패치와 최신 기능들이 반영되어 있습니다.
- 자동 설정 기능과 다양한 내장 기능 덕분에 복잡한 설정 없이도 빠르게 애플리케이션을 개발할 수 있으며, 마이크로서비스 아키텍처 구축에 유리합니다.
- RESTful API, 데이터 액세스, 보안 등의 기능이 통합되어 있어 개발자가 비즈니스 로직에 집중할 수 있는 환경을 제공합니다.

### 3️⃣ SpringData JPA

- Spring Data JPA는 데이터베이스와의 인터랙션을 단순화하고, 불필요한 보일러플레이트 코드를 줄여 개발 효율성을 높여줍니다.

### 5️⃣ PostgreSQL

- PostgreSQL는 복잡한 쿼리 처리와 대규모 데이터셋 관리에 강점을 가집니다.

### 6️⃣ Docker compose

- Docker Compose는 여러 컨테이너를 손쉽게 구성하고 관리할 수 있도록 도와주어, 개발 및 배포 과정에서 환경 설정을 간소화합니다.

### 7️⃣ NCP(CLOVA studio, Server, Object Storage)

- NCP(네이버 클라우드 플랫폼)는 CLOVA Studio 및 서버 인프라 호스팅에 활용되어, 안정적이고 안전한 클라우드 환경을 제공함으로써 프로젝트의 운영 효율성을 높입니다.

### 8️⃣ Hashicorp Vault

- 민감한 데이터(플랫폼 별 사용자 accessToken 등)를 안전하게 관리하고, 보안성을 강화할 수 있습니다.

### 9️⃣ Terraform

- 인프라를 코드로 관리할 수 있게 해 주어, 반복 가능하고 일관된 인프라 구축 및 유지보수가 가능합니다.

### 9️⃣ Promtail, Loki, Prometheus, Grafana

- **Promtail → Loki**: 애플리케이션·시스템 로그를 수집·라벨링해 Loki에 전송 → 대량 로그를 효율적으로 인덱싱·검색합니다.
- **Prometheus**: Pull 방식으로 API 응답 시간·CPU·메모리 같은 시계열 메트릭을 스크랩·저장합니다.
- **Grafana**: Loki·Prometheus 데이터를 대시보드로 시각화합니다.

### 🔟 K6

- JavaScript로 실제 사용자 흐름(로그인→API 호출→스케줄링)을 스크립트화해 REST 성능을 측정합니다.
- `vus`, `stages`, `thresholds` 등 옵션으로 동시 사용자 수·스파이크·지속 테스트를 커스터마이징하면서 테스트합니다.
- 콘솔·JSON 출력으로 응답 시간·처리량·에러율 메트릭을 수집하고 Grafana에 연동해 결과 시각화합니다.

| convention type | description |
| --- | --- |

### 테스트 결과 분석
- 현재 시스템은 동시 약 `1,000건` 수준까지는 안정적으로 요청을 처리할 수 있는 것으로 보입니다. **시나리오 ③**처럼 사용자 수가 점차 증가하는 상황에서도 평균 응답 시간은 `1.94초`, 최대 응답 시간은 `7.88초`로, 대부분의 요청이 정상적으로 처리되었습니다.
- 하지만 **시나리오 ②**처럼 `2,000명`의 동시 요청이 들어오면 평균 응답 시간이 `7.74초`, 최대 `18.28초`까지 증가하면서 응답 지연이 발생하였습니다. 이 결과는 대규모 트래픽에 대한 성능 한계가 있음을 보여주며, 추후 이를 개선할 필요가 있습니다.
- **시나리오 ④**에서는 `1000명`의 사용자가 동시에 요청을 보낸 경우, 총 `2,002건`의 요청이 `5초` 이내에 처리되었습니다. 이는 현재 시스템이 실시간 대응보다는 예약 처리에 더 적합한 구조임을 보여줍니다.
- 일반적으로 사용자 이탈이 늘기 시작하는 5초 이내 응답을 기준으로 예상 접속자 수 약 `1,000명` 정도에 대해서는 충분히 안정적인 성능을 제공할 수 있다고 판단됩니다.
